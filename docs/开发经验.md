## 1. X耗能卡牌在OnPlay中获取X值

**用户需求：** 创建X耗能卡牌，在打出时需要根据消耗的能量数量（X值）产生效果

**实现方案：** 在 `OnPlay` 方法中调用 `ResolveEnergyXValue()` 获取X值。该方法返回玩家实际消耗的能量数，并会经过遗物（如化学X）的修正。不要尝试从 `CardPlay` 对象上获取X值，`CardPlay` 没有 `EnergyCostX` 属性。

```csharp
protected override async Task OnPlay(PlayerChoiceContext choiceContext, CardPlay cardPlay)
{
    int xValue = ResolveEnergyXValue();
    // 使用 xValue ...
}
```

**方案来源：** 查找杀戮尖塔2游戏代码（参考原版X费卡牌如 `Tempest.cs`、`MultiCast.cs`、`Whirlwind.cs` 等），构建发现 `CardPlay.EnergyCostX` 不存在后定位到正确方法

## 2. 多人游戏限定卡牌

**用户需求：** 创建仅在多人游戏模式下可用的卡牌

**实现方案：** 重载 `MultiplayerConstraint` 属性，返回 `CardMultiplayerConstraint.MultiplayerOnly`。该属性在卡牌开发skill中未提及，但原版联机卡牌（如 `BelieveInYou`、`HuddleUp`、`EnergySurge` 等）均使用此属性。

```csharp
public override CardMultiplayerConstraint MultiplayerConstraint => CardMultiplayerConstraint.MultiplayerOnly;
```

需要引入命名空间 `MegaCrit.Sts2.Core.GameActions.Multiplayer`。

**方案来源：** 查找杀戮尖塔2游戏代码（参考原版联机卡牌 `BelieveInYou.cs`、`HuddleUp.cs` 等）

## 3. 本地化描述中energyIcons的正确用法

**用户需求：** 在卡牌/能力本地化描述中显示能量图标

**实现方案：** `energyIcons` 有两种用法：

1. **显示固定N个能量图标**：`{energyPrefix:energyIcons(N)}`，必须使用 `energyPrefix` 作为前缀（固定写法），N为显示的能量图标数量
2. **基于变量值动态显示**：`{VarName:energyIcons()}`，无参形式，根据变量值显示对应数量的能量图标。VarName可以是任意变量名（能力层数可使用 `Amount`），但 `energyIcons()` 不能包含参数。代码中需使用对应类型的变量（如 `EnergyVar`）

```
固定3个图标：{energyPrefix:energyIcons(3)}
基于变量值：{Energy:energyIcons()}  — 需搭配 new EnergyVar(2)
能力层数显示：{Amount:energyIcons()}  — 基于能力层数显示
```

常见错误：
- 在 `energyIcons()` 中传入固定参数如 `energyIcons(1)` —— 无法显示图标，固定数量图标请使用 `energyPrefix`
- 显示固定数量时使用变量名代替 `energyPrefix` —— 前缀必须为 `energyPrefix`

**方案来源：** 用户提示

## 4. 卡牌图片目录与命名规则

**用户需求：** 了解卡牌图片文件的存放目录和命名规则，确保图片路径与代码生成的ID匹配

**实现方案：** 卡牌的 `Id.Entry` 会自动将类名转换为下划线写法，例如 `SeleeDaJi` 的ID为 `selee_da_ji`，而非简单的 `ToLowerInvariant()` 结果 `seleedaji`。`SeleeCard.cs` 中的 `CustomPortraitPath` 和 `PortraitPath` 使用 `Id.Entry.RemovePrefix().ToLowerInvariant()` 生成路径，最终结果已包含下划线。

卡牌图片需要放置在两个目录中：

| 目录 | 对应属性 | 用途 |
|------|----------|------|
| `Selee/images/card_portraits/` | `PortraitPath`（`.CardImagePath()`） | 小尺寸卡牌图，用于卡牌列表等 |
| `Selee/images/card_portraits/big/` | `CustomPortraitPath`（`.BigCardImagePath()`） | 大尺寸卡牌图，用于卡牌详情查看 |

两个目录中的图片文件名必须一致，命名规则为类名转下划线写法再加 `.png`：

```
类名 LiangZiHua     → 图片 liang_zi_hua.png
类名 TanSuoGongJi   → 图片 tan_suo_gong_ji.png
类名 SeleeDaJi      → 图片 selee_da_ji.png
类名 SeleeFangYu    → 图片 selee_fang_yu.png
```

新增卡牌时，需将图片同时拷贝到以上两个目录并按规则命名。

**方案来源：** 用户提示

## 5. 卡牌的ExtraHoverTips根据设计文档的"能力提示"列添加

**用户需求：** 卡牌悬浮时需要显示对应能力的说明提示

**实现方案：** 重载 `ExtraHoverTips` 属性，使用 `HoverTipFactory.FromPower<T>()` 为卡牌添加能力提示。需引入命名空间 `MegaCrit.Sts2.Core.HoverTips`。

基于设计文档开发卡牌时，HoverTips应根据设计文档的**"能力提示"列**添加，而非为所有施加的能力都添加提示。例如"开启门扉"施加了虚弱和易伤，但"能力提示"列为"虚弱、易伤"，两者一致；而"坍缩攻击"虽涉及量子叠加能力，但"能力提示"列为空，因此无需添加量子叠加的HoverTip。

```csharp
protected override IEnumerable<IHoverTip> ExtraHoverTips => [HoverTipFactory.FromPower<LiangZiDieJiaPower>()];
```

其他 `HoverTipFactory` 常用方法：
- `HoverTipFactory.FromKeyword(CardKeyword.Exhaust)` — 关键字提示
- `HoverTipFactory.FromCard<Shiv>()` — 生成卡牌提示
- `HoverTipFactory.Static(StaticHoverTip.Block)` — 静态提示（格挡、召唤等）
- `base.EnergyHoverTip` — 能量图标提示

**方案来源：** 用户提示

## 6. 卡牌描述中使用换行符换行

**用户需求：** 卡牌效果描述中需要换行显示（如"造成6点伤害"和"叠加：额外造成10点伤害"分两行显示）

**实现方案：** 在本地化JSON的描述文本中使用 `\n` 换行符。卡牌开发skill的模板和示例中未提及换行的写法。

```json
"SELEE-TAN_SUO_GONG_JI.description": "造成{Damage:diff()}点伤害。\n[gold]叠加[/gold]：额外造成{DieJiaDamage:diff()}点伤害。"
```

**方案来源：** 用户提示

## 7. 检测能力存在与减少能力层数

**用户需求：** 实现"叠加"效果——当玩家拥有量子叠加能力时触发额外效果，触发后减少一层量子叠加

**实现方案：**

**检测能力是否存在：** 使用 `Creature.HasPower<T>()` 返回 bool，或 `Creature.GetPower<T>()` 获取能力实例（不存在则返回 null）。

```csharp
if (base.Owner.Creature.HasPower<LiangZiDieJiaPower>())
{ ... }

var power = base.Owner.Creature.GetPower<LiangZiDieJiaPower>();
if (power != null)
{ ... }
```

**减少能力层数：** 使用 `PowerCmd.Decrement(power)` 将能力层数减1。若减至0且能力为 Counter 类型，会自动移除。也可使用 `PowerCmd.ModifyAmount()` 减少任意层数，或 `PowerCmd.Remove<T>(creature)` 直接移除整个能力。

```csharp
var dieJiaPower = base.Owner.Creature.GetPower<LiangZiDieJiaPower>();
if (dieJiaPower != null)
{
    // 触发叠加效果...
    await PowerCmd.Decrement(dieJiaPower);
}
```

**方案来源：** 查找杀戮尖塔2游戏代码（参考 `Dismantle.cs` 的 `HasPower` 用法、`BurstPower.cs` 和 `OneTwoPunchPower.cs` 的 `PowerCmd.Decrement` 用法、`Expose.cs` 的 `PowerCmd.Remove` 用法）

## 8. 卡牌条件发光（ShouldGlowGoldInternal）

**用户需求：** 当玩家拥有特定能力时，卡牌在手牌中金色高亮显示，提示玩家该卡牌可触发额外效果

**实现方案：** 重载 `ShouldGlowGoldInternal` 属性，返回 true 时卡牌在手牌中会发出金色光芒。常用于条件触发效果的卡牌，帮助玩家识别何时能触发额外效果。

```csharp
protected override bool ShouldGlowGoldInternal =>
    base.Owner != null && base.Owner.Creature.HasPower<LiangZiDieJiaPower>();
```

注意需要做空值检查（`base.Owner != null`），因为在某些上下文（如卡牌预览）中 `Owner` 可能为 null。

**方案来源：** 查找杀戮尖塔2游戏代码（参考 `Dismantle.cs`、`BubbleBubble.cs` 等卡牌的 `ShouldGlowGoldInternal` 实现）

## 9. "下回合获得能力"的实现方式

**用户需求：** 创建"下回合获得N层量子叠加"效果的卡牌，能力在下一回合开始时才施加给玩家

**实现方案：** 创建一个对应的"NextTurn"版本的能力类，继承 `SeleePower`，在 `AfterSideTurnStart` 钩子中施加实际能力并移除自身。参考原版 `EnergyNextTurnPower`、`DrawCardsNextTurnPower` 等的模式。

```csharp
public class LiangZiDieJiaNextTurnPower() : SeleePower
{
    public override PowerType Type => PowerType.Buff;
    public override PowerStackType StackType => PowerStackType.Counter;

    public override async Task AfterSideTurnStart(CombatSide side, IReadOnlyList<Creature> participants, ICombatState combatState)
    {
        if (participants.Contains(base.Owner) && base.AmountOnTurnStart != 0)
        {
            Flash();
            await PowerCmd.Apply<LiangZiDieJiaPower>(new ThrowingPlayerChoiceContext(), base.Owner, base.Amount, base.Owner, null);
            await PowerCmd.Remove(this);
        }
    }
}
```

关键点：
- 使用 `AfterSideTurnStart` 而非 `AfterSideTurnEnd`，确保在回合开始时触发
- 使用 `base.AmountOnTurnStart != 0` 防止重复触发
- 在 `PowerCmd.Apply` 中使用 `ThrowingPlayerChoiceContext` 作为上下文（因为在能力钩子中没有现成的 `PlayerChoiceContext`）
- 两个能力（NextTurn版和实际版）都需要在本地化中添加条目，且NextTurn版也需要 `smartDescription`

**方案来源：** 查找杀戮尖塔2游戏代码（参考 `EnergyNextTurnPower.cs`、`DrawCardsNextTurnPower.cs` 的实现模式）

## 10. Power中Owner的类型与Card中不同

**用户需求：** 在Power类中使用 `PowerCmd.Apply` 施加另一个能力时，确定正确的目标参数

**实现方案：** 在Power类中，`base.Owner` 的类型已经是 `Creature`，无需再访问 `.Creature` 属性。这与Card类中 `base.Owner.Creature` 的写法不同。

```csharp
// Card中（Owner是CardOwner类型，需要.Creature）
await PowerCmd.Apply<SomePower>(choiceContext, base.Owner.Creature, amount, base.Owner.Creature, this);

// Power中（Owner已经是Creature类型，直接使用）
await PowerCmd.Apply<SomePower>(new ThrowingPlayerChoiceContext(), base.Owner, amount, base.Owner, null);
```

**方案来源：** 构建发现问题（编译错误 `Creature未包含Creature的定义`）

## 11. CreatureCmd.GainBlock的重载选择

**用户需求：** 在卡牌中需要根据条件动态计算格挡值后调用 `CreatureCmd.GainBlock`

**实现方案：** `CreatureCmd.GainBlock` 有两个重载：

| 重载 | 签名 | 适用场景 |
|------|------|----------|
| BlockVar版 | `GainBlock(Creature, BlockVar, CardPlay?)` | 直接使用卡牌的BlockVar（标准格挡） |
| decimal版 | `GainBlock(Creature, decimal, ValueProp, CardPlay?)` | 需要动态计算格挡值时 |

当需要动态修改格挡值（如叠加效果额外增加格挡）时，应使用第二个重载，传入 `ValueProp.Move` 作为属性参数：

```csharp
decimal block = base.DynamicVars.Block.BaseValue;
// ... 动态修改 block ...
await CreatureCmd.GainBlock(base.Owner.Creature, block, ValueProp.Move, cardPlay);
```

**方案来源：** 构建发现问题（编译错误 `无法从decimal转换为BlockVar`）

## 12. 叠加效果中能力层数减少的时序

**用户需求：** 具有"叠加"效果的卡牌，在触发叠加额外效果后减少量子叠加层数，需要确保层数减少在效果结算完成后触发

**实现方案：** `PowerCmd.Decrement` 应放在效果执行之后，而非之前。如果先减少层数再执行效果，可能导致其他依赖该能力层数的效果计算不正确。

```csharp
// 错误：先减少层数再执行效果
var dieJiaPower = base.Owner.Creature.GetPower<LiangZiDieJiaPower>();
if (dieJiaPower != null)
{
    damage += DynamicVars["DieJiaDamage"].BaseValue;
    await PowerCmd.Decrement(dieJiaPower);  // 先减少
}
await DamageCmd.Attack(damage)...  // 后执行效果

// 正确：先执行效果再减少层数
var dieJiaPower = base.Owner.Creature.GetPower<LiangZiDieJiaPower>();
if (dieJiaPower != null)
{
    damage += DynamicVars["DieJiaDamage"].BaseValue;
}
await DamageCmd.Attack(damage)...  // 先执行效果

if (dieJiaPower != null)
{
    await PowerCmd.Decrement(dieJiaPower);  // 后减少
}
```

注意：需要在效果执行前先获取能力引用并判断是否存在，将结果保存到局部变量，效果执行后再根据该变量决定是否减少层数。

**方案来源：** 用户提示

## 13. Python脚本中文输出乱码问题

**用户需求：** 通过Bash工具执行Python脚本（如设计文档读取脚本）时，输出中的中文字符显示为乱码

**实现方案：** Windows PowerShell 5.1 的标准输出编码默认为系统ANSI代码页（中文Windows为GBK/GB2312），而非UTF-8。Bash工具捕获输出后按UTF-8解读，导致乱码。在Python脚本开头添加 `sys.stdout.reconfigure(encoding='utf-8')` 强制stdout使用UTF-8编码即可解决。

```python
import sys
sys.stdout.reconfigure(encoding='utf-8')
```

其他无效的尝试（均已验证无法解决）：
- `chcp 65001` — 改变控制台代码页，不影响Python进程的stdout编码
- `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8` — 改变PowerShell的输出编码，但Python的stdout编码在进程启动时已确定
- 将输出重定向到UTF-8文件再用Read工具读取 — 可行但繁琐，每次需额外步骤

**方案来源：** 构建发现问题（执行 `read_design_doc.py` 输出乱码，尝试多种方案后定位到Python stdout编码问题）

## 14. 叠加效果必须通过SeleeHook.AfterDieJiaTrigger触发

**用户需求：** 具有"叠加"额外效果的卡牌，在触发叠加效果后需要减少量子叠加层数，同时通知其他监听者（如"蝶舞"卡牌）

**实现方案：** 所有含有叠加额外效果的卡牌，在叠加效果执行完毕后，必须调用 `SeleeHook.AfterDieJiaTrigger(Owner, this)` 而非直接调用 `PowerCmd.Decrement`。`SeleeHook.AfterDieJiaTrigger` 内部会遍历所有 `ISeleeHook` 监听者通知叠加事件，然后再自动执行 `PowerCmd.Decrement` 减少量子叠加层数。

```csharp
// 错误：直接调用PowerCmd.Decrement
if (dieJiaPower != null)
{
    // 执行叠加额外效果...
    await PowerCmd.Decrement(dieJiaPower);
}

// 正确：通过SeleeHook触发，让监听者响应后再减少层数
if (dieJiaPower != null)
{
    // 执行叠加额外效果...
    await SeleeHook.AfterDieJiaTrigger(Owner, this);
}
```

关键点：
- `SeleeHook.AfterDieJiaTrigger` 的参数为 `(Player owner, CardModel? triggerCard)`，在卡牌中传入 `Owner` 和 `this`
- 该方法会先通知所有实现 `ISeleeHook` 的监听者，最后自动执行 `PowerCmd.Decrement` 减少一层量子叠加
- 监听者可以通过 `triggerCard` 参数判断是哪张卡牌触发的叠加效果

**方案来源：** 用户提示

## 15. 卡牌实现ISeleeHook被动响应叠加事件

**用户需求：** 创建如"蝶舞"这样的卡牌——当其他卡牌触发叠加效果时，被动地将自身加入手牌

**实现方案：** 卡牌可以实现 `ISeleeHook` 接口，通过 `AfterDieJiaTrigger` 方法被动响应叠加事件。游戏引擎的 `IterateHookListeners()` 会遍历战斗中所有对象（包括所有牌堆中的卡牌），因此卡牌在任何牌堆中都能接收到事件通知。

```csharp
public class DieWu() : SeleeCard(0, CardType.Attack, CardRarity.Uncommon, TargetType.AnyEnemy), ISeleeHook
{
    public override IEnumerable<CardKeyword> CanonicalKeywords => [SeleeCardKeyword.DieJia];

    public async Task AfterDieJiaTrigger(Player owner, CardModel? triggerCard)
    {
        if (Pile?.Type != PileType.Hand)
        {
            await CardPileCmd.Add(this, PileType.Hand);
        }
    }

    protected override async Task OnPlay(PlayerChoiceContext choiceContext, CardPlay cardPlay)
    {
        // 正常打出效果，不含叠加逻辑
    }
}
```

关键点：
- `ISeleeHook.AfterDieJiaTrigger` 返回 `Task`（异步方法），支持在回调中执行异步操作
- "将这张牌放入手牌"指的是将卡牌自身移入手牌，而非生成复制。使用 `CardPileCmd.Add(this, PileType.Hand)` 实现
- 需要判断卡牌当前不在手牌中时才移入，使用 `Pile?.Type != PileType.Hand` 检查
- 只要卡牌描述中提到叠加效果，就应当添加 `DieJia`（叠加）关键字，以便玩家在游玩时看到对应的说明文本提示
- 需要引入命名空间 `MegaCrit.Sts2.Core.Models`（`CardModel` 所在命名空间）和 `MegaCrit.Sts2.Core.Entities.Players`（`Player` 所在命名空间）

**方案来源：** 用户提示

## 16. 能力效果数值应使用Amount记录以保证可叠加

**用户需求：** 创建能力牌（Power卡牌）对应的能力效果，需要支持多张同名卡牌打出后效果叠加

**实现方案：** 能力效果的数值应使用 `base.Amount`（能力层数）记录，而非 `DynamicVar`。这样当多次打出同名卡牌时，`PowerCmd.Apply` 会自动叠加 Amount 层数，效果也随之叠加。如果使用 `DynamicVar` 记录数值，再次施加时 DynamicVar 不会叠加，会导致效果丢失。

**卡牌侧：** 使用 `PowerVar<T>` 声明变量，这样卡牌描述中能正确显示数值，`OnPlay` 时以 DynamicVar 值作为 Amount 施加能力：

```csharp
// 卡牌中
protected override IEnumerable<DynamicVar> CanonicalVars => [
    new PowerVar<SomePower>(20),
];

protected override async Task OnPlay(PlayerChoiceContext choiceContext, CardPlay cardPlay)
{
    await PowerCmd.Apply<SomePower>(choiceContext, base.Owner.Creature, DynamicVars["SomePower"].BaseValue, base.Owner.Creature, this);
}

protected override void OnUpgrade()
{
    DynamicVars["SomePower"].UpgradeValueBy(10m);
}
```

**能力侧：** 移除 `CanonicalVars` 中的对应变量，效果中直接使用 `base.Amount`：

```csharp
// 能力中 - 错误：使用DynamicVar
protected override IEnumerable<DynamicVar> CanonicalVars => [
    new DynamicVar("Value", 20m),
];
// 触发效果时使用 DynamicVars["Value"].BaseValue → 再次施加时不叠加！

// 能力中 - 正确：使用Amount
// 无需声明CanonicalVars
// 触发效果时使用 base.Amount → 再次施加时Amount叠加，效果随之叠加
await PowerCmd.Apply<OtherPower>(choiceContext, base.Owner, base.Amount, base.Owner, null);
```

**本地化：** `smartDescription` 中使用 `{Amount}` 替代 `{VarName:diff()}` 显示数值。

关键点：
- 此规则适用于能力牌（Power类型卡牌）对应的可叠加能力。对于技能牌/攻击牌施加的一次性能力，仍可使用 DynamicVar
- `PowerVar<T>` 会自动以能力类名作为变量键名（如 `PowerVar<SomePower>` → 键名 `"SomePower"`）
- 卡牌升级时通过 `DynamicVars["SomePower"].UpgradeValueBy()` 增加数值，影响的是后续打出时的 Amount 值

**方案来源：** 用户提示（星环盈余卡牌开发中发现DynamicVar方案不可叠加，改为Amount方案）

## 17. 能力效果实现优先参考设计文档能力sheet的描述

**用户需求：** 实现能力效果时，确定效果的实现方式

**实现方案：** 实现能力效果时，**应优先查询设计文档能力sheet页的"说明"和"备注"列**。设计文档中通常会包含能力的具体行为描述、触发条件、实现提示等关键信息。仅当设计文档无相关描述时，再参考现有模组能力或游戏本体能力的实现方式。

查询设计文档能力sheet的方式：

```bash
python .opencode/scripts/read_design_doc.py 能力 {能力名称}
```

返回结果包含"名称"、"说明"、"叠加方式"、"类型"、"备注"等字段。其中：

- **说明**列：能力的核心效果描述，通常包含触发条件、效果逻辑等，是实现的主要依据
- **备注**列：补充的实现细节，如特殊实现方式（`通过ISeleeHook.ModifyDieJiaBlockAdd实现`）、参考对象（`可参考遗物"恶魔之舌"的实现`）、特殊属性（`重复获得时生成独立能力；public override PowerInstanceType InstanceType => PowerInstanceType.Instanced;`）等

**示例：** 设计文档中"星环盈余"的说明为"进入星环爆发状态时，获得X层星环能级\n星环爆发状态下再次进入时不触发"，备注为空。据此可直接确定实现方式为监听 `AfterPowerAmountChanged` 检测 `XingHuanBaoFaPower` 的施加，并通过 `power.Amount == amount` 判断是否为首次进入。
